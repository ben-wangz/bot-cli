# image-gen-cli 设计方案（v0）

## 1. 目标与范围

`image-gen-cli` 是一个面向 agent/CLI 场景的图片生成工具，第一阶段只支持**文生图**，底层固定对接 `gpt-image-2`（按 `todo/API_IMAGE2_接口说明.md` 约定）。

本阶段目标：

- 提供稳定、可脚本化的图片生成能力。
- 与 Responses + `image_generation` tool 协议对齐，正确处理 streaming 与 non-streaming。
- 保证“最终图结果”可可靠落盘与追踪，不把预览图当成最终结果。

本阶段不做：

- 多后端适配抽象（暂不支持可插拔 provider）。
- 图生图、多图编辑、风格迁移等扩展能力。
- 复杂会话编排产品化（仅保留必要的上下文续链能力）。

---

## 2. 用户与使用场景

### 2.1 核心用户

- 在终端里快速生成图片的开发者。
- 在自动化流程里批量出图的脚本/CI 使用者。
- 需要 agent 可调用能力的工作流用户。

### 2.2 典型场景

- 基础文生图：输入一段描述生成单张图并保存到本地。
- 参数化出图：指定尺寸、质量、格式、背景等参数以满足不同用途。
- 流式观察：开启 streaming 观察生成进度（可选预览），完成后保存最终图。
- 会话续链：基于上一轮响应 ID 在同一上下文继续出图（`store` + `previous_response_id`）。
- 续链降级：当后端不支持 `store/previous_response_id` 时，明确提示并退化为无状态请求。

---

## 3. 设计原则

- **最终结果优先**：以 completed 响应中的 `output[].result` 作为唯一最终图来源。
- **协议对齐优先**：尽量贴近 API 规范，减少“魔改语义”。
- **可观测优先**：命令输出应清晰区分“预览中”“已完成”“失败”。
- **默认可用**：提供合理默认参数，降低首次使用门槛。
- **渐进扩展**：当前实现不做多后端抽象，但保留未来扩展接口边界。

---

## 4. 功能范围（v0）

### 4.1 必备能力

- 文本提示词生成图片（单次请求）。
- 生成参数控制：
  - 尺寸（size）
  - 质量（quality）
  - 输出格式（output_format）
  - 压缩质量（output_compression，适用于可压缩格式）
  - 背景（background）
- 输出文件保存到本地路径。
- 支持 streaming/non-streaming 两种请求模式。
- 支持最小化上下文续链（`store` + `previous_response_id`）。

### 4.2 结果与元信息输出

- 命令完成后输出：
  - 任务状态（成功/失败）
  - 输出文件路径
  - 关键请求参数快照
  - 响应 ID（用于续链）
- 可选输出结构化结果，便于脚本消费。

### 4.3 错误处理

- 参数无效：本地快速失败并提示可选值范围。
- 上游失败：透传关键信息（错误码、错误消息）。
- streaming 过程中异常中断：明确标记失败，不产出伪最终图。

---

## 5. 关键交互设计

### 5.1 生成流程

1. 读取输入与配置。
2. 组装请求并发起生成。
3. 若为 streaming：
   - 接收进度事件；
   - 如有预览图事件，仅用于过程展示。
4. 接收完成事件并提取最终图片结果。
5. 落盘并输出执行结果。

### 5.2 预览与最终图边界

- `partial_image_b64` 只用于“生成中预览”。
- 最终保存必须来自 `image_generation_call.result`。
- 若没有最终 `result`，即使出现 completed 迹象，也按失败处理。

### 5.3 会话续链

- 当后端确认支持 `store` 与 `previous_response_id` 时，允许续链。
- 若后端不支持，CLI 不伪造续链语义，应明确提示并退化到无状态请求。

---

## 6. 配置与参数策略

### 6.1 配置来源优先级

建议采用以下优先级（高到低）：

1. 命令行显式参数
2. 环境变量
3. 内置默认值

本阶段配置不复杂，**不引入配置文件**。所有能力通过命令参数和环境变量即可覆盖。

建议支持的配置项：

- 连接与鉴权：
  - API Key
  - API Base URL（用于兼容不同网关/中转域名）
- 生成参数：
  - `model`（Responses 主模型）
  - `image_model`（固定默认 `gpt-image-2`，允许显式声明）
  - `size`
  - `quality`
  - `output_format`
  - `output_compression`
  - `background`
- 请求行为：
  - `stream`（streaming 或 non-streaming）
  - `store`
  - `previous_response_id`
- 输出行为：
  - 输出目录/输出文件名（或命名规则）
  - 机器可读输出开关（用于脚本化）
  - 进度展示开关（交互模式开，脚本模式可关）

### 6.2 默认值策略

- 默认采用“质量与速度平衡”的参数组合。
- 默认输出到当前工作目录下可识别文件名。
- 默认启用对用户友好的进度反馈（脚本模式可关闭）。
- 默认不强制开启 `store`，由用户按是否需要续链显式控制。

### 6.3 参数合法性

- 对枚举值做本地校验（如格式、质量、背景）。
- 对尺寸做白名单或受控集合校验。
- 对压缩参数做范围校验与格式联动校验。

---

## 7. 输出与可脚本化

面向两类输出：

- 人类可读输出：简洁状态 + 路径 + 关键摘要。
- 机器可读输出：稳定字段结构，便于管道和自动化解析。

行为约束：

- 成功时必须可直接拿到最终文件路径。
- 失败时必须返回非成功退出语义，并包含可定位原因。

---

## 8. 安全与合规考虑

- API Key 只通过安全渠道输入（环境变量/受控配置），不在日志明文展示。
- 日志默认不记录完整图片 base64。
- 对外输出避免泄漏敏感请求头与凭证信息。

---

## 9. 测试与验收标准（功能层）

采用一个最小端到端正常用例完成验收，总计两次生图调用：

1. 第一次调用使用 streaming 模式生成第 1 张图，并保存最终图。
2. 第二次调用使用 non-streaming 模式，并携带第一次返回的 `response_id` 进行续链，生成第 2 张图。

通过条件：

- 两次调用都成功，且各自落盘为可打开图片文件。
- 第一次调用可观察到 streaming 过程（如有预览仅作预览）。
- 第二次调用确实使用续链参数完成请求。
- 两次最终保存都来自最终 `output[].result`，不存在用预览图替代最终图的情况。

---

## 10. 里程碑建议

### M1（最小验收链路）

- 完成第一次生图：streaming 请求、过程可观测、最终图落盘。
- 完成第二次生图：non-streaming 请求，携带 `previous_response_id` 续链并落盘。
- 两次调用统一以最终 `output[].result` 作为保存来源。

### M2（收口与稳定）

- 完成最小端到端用例的回归固化（仅上述两次调用链路）。
- 完成结果输出收口（成功/失败状态、文件路径、response id）。
- 完成必要的参数校验与错误提示收口，避免假成功。

---

## 11. 后续扩展方向（暂不实现）

- 图生图（引用本地图片或远程图片作为输入源）。
- 多图批量任务与并发控制。
- 风格模板与可复用 prompt 资产。
- 成本估算与调用统计展示。

该扩展不影响 v0 架构落地；v0 优先保证“单后端、单能力、稳定结果”。
