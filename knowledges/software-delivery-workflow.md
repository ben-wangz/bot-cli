# 软件开发流程抽象（需求 -> 设计 -> 分阶段 Issue）

- 状态: living
- 适用范围: 本仓库后续所有软件开发任务
- 维护方式: 随项目推进持续增量修订

## 1. 目标

把一次完整开发过程抽象为可复用流程，减少沟通成本、避免遗漏，并让每次迭代都能沉淀到统一方法论。

## 2. 核心原则

1. 单一信息源：执行层以 issue 文档为准，避免多套并行清单。
2. 先约束后实现：先确认边界、资源和验收，再开始编码。
3. 先主流程后分支：先打通正向主路径，再补异常与优化。
4. 可复盘：每个阶段必须有输入、输出、验收标准和未决项。
5. 可演进：流程文档是活文档，按实践不断修正。

## 3. 标准流程（7 步）

## Step 1 - 接收与澄清需求

目标：把“想做什么”转换成“可执行范围”。

产出：

1. 目标定义（业务目标 + 技术目标）。
2. 范围边界（In Scope / Out of Scope）。
3. 已知约束（权限、环境、资源、时段、安全要求）。
4. 成功标准（最终如何判断完成）。

检查点：

- 需求中模糊项是否被显式澄清。
- 不可做事项是否明确写出。

## Step 2 - 方案设计与技术分层

目标：把需求映射为架构与模块责任。

产出：

1. 总体设计（命令层/应用层/基础设施层等）。
2. 关键技术决策（语言、协议、认证、状态管理）。
3. 统一执行模型（输入/输出/诊断结构）。
4. 关键风险与缓解策略。

检查点：

- 关键约束是否在设计中可落地（不是口号）。
- 是否存在“实现后才发现权限/环境不支持”的风险。

## Step 3 - 阶段化拆解（Phase Planning）

目标：按依赖关系拆成可独立评审的阶段。

产出：

1. Phase 顺序（例如 P0 -> P1 -> P2 ...）。
2. 每阶段范围（功能点、接口、测试范围）。
3. 每阶段验收条件（完成定义 DoD）。

检查点：

- 阶段边界是否清晰，是否可单独验收。
- 依赖是否单向，避免循环依赖。

说明：

- 这里可以先内部阶段化思考，但不要求长期保留独立 phase 文档。
- 一旦进入执行，统一沉淀到 issue 文档，避免重复维护。

## Step 4 - 形成执行清单（Issue 化）

目标：把阶段计划变成可执行任务。

产出：

1. Issue 总看板（tracking board）。
2. 每阶段 1 个主 issue（可再细分子任务）。
3. 每个 issue 包含：目标、范围、任务清单、依赖、验收标准。

检查点：

- 每个 issue 是否可“拿来就做”。
- issue 是否可独立评审（不依赖口头上下文）。

## Step 5 - 测试策略先行

目标：在开发前约定验证路径。

产出：

1. 测试分层（单测/集成/环境/E2E）。
2. 正向主流程用例定义。
3. 自动化执行方式（例如 OpenCode prompt）。
4. 清理策略（测试资产回收规则）。

当前已沉淀约束（可复用）：

1. 每个 action 仅 1 条独立正向测试用例。
2. 每条 action 用例必须严格独立，不能依赖执行顺序或其他用例输出。
3. 用例所需动态输入（如 node/vmid/upid）必须在单条用例内自解析，不跨用例传值。
4. 对有业务依赖的 action，可在前置条件声明“依赖能力已通过”，但执行数据仍需本用例内闭环。
5. E2E 用例独立存在，不替代 action 级验证。
6. 若使用 sub-agent 并发执行测试套件，需显式设置并发上限（默认建议 <= 2）。
7. 代码侧测试工件目录（如 `src/test/prompts/`）中的 prompt 文案统一使用英文。
8. 对共享基础设施的写操作测试（如 VM 生命周期），应采用“读可并发、写串行”策略，限制有效写并发为 1。
9. 每条写操作用例必须在用例内申请独立资源标识（如 fresh VMID），并在成功/失败路径都执行回收。
10. 一旦发现某个测试受到其他测试影响，应判定为“测试不独立”缺陷；必须先定位共享状态/资源耦合根因并修复，再继续放量执行。
11. 对 PVE 类环境，建议通过 `PVE_ALLOWED_VMID_MIN` / `PVE_ALLOWED_VMID_MAX` 控制可操作 ID 范围；默认建议 `1001-2000`。

## Step 6 - 执行与阶段评审

目标：按 issue 推进并阶段性复盘。

执行规范：

1. 每完成一个任务，立即更新 issue 状态。
2. 偏离计划时，先更新 issue 再继续实现。
3. 每阶段结束做一次短复盘：完成项、问题、调整项。

评审输出：

1. 已完成清单。
2. 未完成原因。
3. 下一阶段入口条件是否满足。

## Step 7 - 沉淀与流程升级

目标：把一次实践转化为长期能力。

产出：

1. 本文档增量更新（新增经验/去除无效步骤）。
2. 模板优化（issue 模板、测试 prompt 模板）。
3. 新的默认约束（如资源边界、清理规则）。

## 4. 工件规范（本仓库）

建议目录：

```text
knowledges/
  software-delivery-workflow.md

applications/<module>/todo/<module>/
  development-plan.md
  issues/
    000-tracking-board.md
    001-*.md
    002-*.md
    ...
```

约定：

1. `development-plan.md` 负责描述总体原则与稳定约束。
2. `issues/` 是执行层唯一清单来源。
3. 避免与 issues 重复的中间清单长期存在。

## 5. Issue 模板（简化版）

```md
# ISSUE-XXX <title>

- status: open
- priority: high|medium|low
- depends_on: ISSUE-...

## Goal

<一句话目标>

## Scope

- <范围1>
- <范围2>

## Tasks

- [ ] <任务1>
- [ ] <任务2>

## Acceptance

- [ ] <验收条件1>
- [ ] <验收条件2>
```

## 6. 反模式（避免）

1. 文档重复：phase 清单和 issue 清单双维护。
2. 未约束先开发：资源/权限/环境不清就开工。
3. 用 E2E 替代基础验证：导致定位成本高。
4. 只写任务不写验收：完成标准不一致。

## 7. 本次案例沉淀（proxmox-cli）

本案例验证了以下流程有效：

1. 先收敛约束（语言、console 模式、测试策略、资源边界）。
2. 再完成方案文档与 issue 化。
3. 最终以 issue 目录作为执行唯一入口，降低维护成本。
4. 引入 suite-runner prompt，明确 sub-agent 模式、并发上限与聚合输出契约。
5. 通过“顺序无关 + 输入自解析”规则，保证并发测试稳定性与可重复执行。

后续每完成一个 phase，建议在本文档新增“实践修订”小节，记录：

1. 哪条规则有效。
2. 哪条规则需要调整。
3. 是否需要新增默认模板。

## 8. 并发测试清单（建议）

在引入 sub-agent 并发前，先逐条检查：

1. 用例是否引用了“上一个用例产物”（若有则不通过）。
2. 用例是否使用固定冲突资源（同名文件、同一 VM、同一端口）。
3. 用例是否具备本地发现输入能力（node/vmid/upid 可在本用例解析）。
4. suite-runner 是否定义并发上限、失败策略和结果聚合格式。
5. 用例文本是否符合仓库语言规范（代码目录默认英文）。
